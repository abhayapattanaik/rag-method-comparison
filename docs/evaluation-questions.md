# Evaluation Questions

## Overview

The evaluation set contains 32 questions derived from 8 source papers. Questions were generated by an LLM (with temperature 0.2 for diversity), then retained as-is after passing the generation pipeline's quality constraints. Each question was written to require genuine understanding of the paper — not simple keyword matching.

The set is designed for scientific comparison: all four retrieval pipelines (Traditional, Contextual, Hybrid, Modern) are evaluated against the same 32 questions, the same ground-truth answers, and the same LLM judge.

**Distribution:** 4 questions per paper, covering 8 papers = 32 questions total.

### Paper 01 — RAG (Lewis et al., 2020)

| Field | Value |
|-------|-------|
| **Question ID** | `8b471c86e7be90ce` |
| **Question** | How do RAG-Sequence and RAG-Token differ in their treatment of retrieved documents during generation, and what implications does this have for tasks like Jeopardy question generation? |
| **Ground Truth** | RAG-Sequence uses the same retrieved document to generate the entire output sequence, treating it as a single latent variable marginalized via a top-K approximation. RAG-Token draws a different latent document for each target token, allowing the generator to combine content from multiple documents within a single answer (p. 3). This distinction matters for Jeopardy question generation, where questions often contain two separate pieces of information; RAG-Token outperforms RAG-Sequence on this task because it can aggregate facts from several retrieved passages, as demonstrated by the Hemingway example where different documents dominate the posterior for different generated book titles (pp. 6–7). |

### Paper 03 — Late Chunking (Gunther et al., 2024)

| Field | Value |
|-------|-------|
| **Question ID** | `29e56478499db50b` |
| **Question** | How does late chunking differ from naive chunking in its processing pipeline, and why does this architectural difference lead to better chunk embeddings? |
| **Ground Truth** | In naive chunking, text is split before being processed by the embedding model, so each chunk is encoded independently without awareness of surrounding text. Late chunking first passes all tokens of the entire document through the transformer to generate contextualized token embeddings, and only then applies chunking boundaries before mean pooling (pp. 3–4). Each token embedding already contains contextual information from the full document, so the resulting chunk embeddings capture long-range semantic dependencies — for example, resolving coreferences like "its" or "the city" back to "Berlin" mentioned in an earlier chunk (pp. 1–2). |
| **Source Pages** | 1, 2, 3, 4 |
| **Type** | Methodology — architectural comparison |

| **Ground Truth** | Averaging results across three embedding models and four BeIR datasets, late chunking achieved a 3.63% relative improvement (1.9% absolute in nDCG@10) over naive chunking with sentence boundaries, a 3.46% relative improvement (1.8% absolute) with fixed-size boundaries, and a 2.70% relative improvement (1.5% absolute) with semantic sentence boundaries (p. 7). Sentence boundaries showed the largest relative gain, and the late chunking counterpart almost always yielded better performance regardless of chunking method. |

| **Ground Truth** | Long late chunking splits the document into larger macro chunks of lmax tokens, each encompassing multiple smaller chunks, and processes each macro chunk separately through the LateChunking method (p. 4). ### Paper 04 — RAPTOR (Sarthi et al., 2024)

| Field | Value |

| **Question ID** | `25289ec84aa738cc` |
| **Question** | How does RAPTOR's clustering algorithm differ from traditional text segmentation approaches, and why is soft clustering with Gaussian Mixture Models preferred over hard clustering? |
| **Ground Truth** | RAPTOR uses soft clustering based on Gaussian Mixture Models (GMMs), where nodes can belong to multiple clusters without requiring a fixed number of clusters. This is essential because individual text segments often contain information relevant to various topics, warranting their inclusion in multiple summaries. | **Question** | How does the hybrid retrieval approach achieve efficiency comparable to a standard lexical-only system, and what specific technology enables the semantic retrieval component to run without adding latency overhead? |
| **Ground Truth** | The hybrid approach achieves efficiency by running semantic and lexical retrieval in parallel rather than sequentially, so no extra overhead is added on top of the standard inverted index-based lexical approach. | **Ground Truth** | The authors rejected using semantic scores for merging because generating semantic scores for lexically-retrieved documents would require running them through the neural model (inefficient), and because semantic retrieval is not as discriminative as lexical retrieval — embeddings act as smoothed representations that struggle to distinguish between semantically similar and exactly matched text. ### Paper 06 — Fusion Functions (Bruch et al., 2022)

| Field | Value |
|-------|-------|
| **Question ID** | `915bbcc710dbd87b` |
| **Question** | Why do the authors argue that Reciprocal Rank Fusion (RRF) loses valuable information compared to convex combination, and what formal property do they use to characterize this information loss? |
| **Ground Truth** | The authors argue that RRF discards information because it operates on ranks rather than raw scores, meaning the distance between scores plays no role in determining the hybrid score. They formalize this using Lipschitz continuity: RRF lacks this property because a small change in individual scores can cause a hard transition in ranks, leading to exaggerated changes in the fused score. They demonstrate this empirically through SRRF, a smooth approximation of RRF using a sigmoid function, showing that reducing the Lipschitz constant (smaller beta) actually improves ranking quality over standard RRF (pp. 19–21). |
| **Source Pages** | 19, 20, 21 |

### Paper 08 — Multi-Stage Ranking (Nogueira et al., 2019)

| Field | Value |
|-------|-------|
| **Question ID** | `f146b5dfeb184f81` |
| **Question** | How do monoBERT and duoBERT differ in their classification approaches to document ranking, and how are they arranged in the multi-stage pipeline? |
| **Ground Truth** | monoBERT is a pointwise re-ranker that treats document ranking as binary classification over individual candidate documents, using the [CLS] vector from BERT to produce a relevance probability for each query-document pair independently. duoBERT is a pairwise re-ranker that estimates the probability of one candidate being more relevant than another by taking a query and two candidates as input. They are arranged in a three-stage pipeline: BM25 retrieves the initial top-k0 candidates (H0), monoBERT re-ranks them and passes top-k1 to duoBERT (H1), and duoBERT produces the final ranking (H2). This design allows increasingly rich models to operate on progressively smaller candidate sets, controlling inference latency (pp. 1–4). |

## Generation Process

Questions were generated using `src/evaluation/question_gen.py` via the CLI.

### How It Works

1. The generator scans `data/papers/extracted/` for `.md` files (one per paper).
2. Questions are distributed roughly evenly: `count / num_papers` per paper (minimum 1).
3. Each paper's full Markdown text is sent to the LLM in a single call with the following prompt constraints:
   - Answerable: the answer must be explicitly stated or implied in the paper.
   - Non-trivial: requires understanding of concepts, methods, or results.
   - Diverse: cover motivation, methodology, experimental results, limitations, comparisons.
   - Specific: reference concrete details (numbers, model names, dataset names).
4. Ground truths must be 2–4 sentences, grounded in specific text, with page number citations.
5. The LLM returns a raw JSON array (no prose). The parser strips accidental markdown fences and skips malformed items.
6. Each question receives a deterministic 16-hex-char ID derived from `SHA-256(question_text)`.
7. Results are saved to `data/questions.json`. The operation is idempotent: if the file already contains `>= count` questions, generation is skipped.

### LLM Parameters

| Parameter | Value |
|-----------|-------|
| Temperature | 0.2 |
| System prompt | Expert research assistant, well-formed JSON only |
| Output format | JSON array with `question`, `ground_truth`, `source_pages` fields |

### Cost Gate

The generator respects the project cost gate. Running without `--approve` prints a token estimate and exits. Running with `--approve` executes the LLM calls.

## How to Regenerate

Check if generation is needed first (idempotent — safe to call even if questions already exist):

```bash
python3 -m src.cli.main generate-questions --count 32
```

To actually run generation:

```bash
python3 -m src.cli.main generate-questions --count 32 --approve
```

Output is written to `data/questions.json`. The file is gitignored because question content is specific to the local paper extractions. Each installation generates its own question set from the same source PDFs.
